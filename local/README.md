# Local development

The full stack runs in Docker — no need to deploy on KTH cloud or be a
Keycloak admin to develop on this fork.

```
┌────────────┐   ┌────────────┐   ┌────────────┐
│  Keycloak  │   │  Postgres  │   │   Garage   │
│   :8081    │   │   :5432    │   │ :3900/3903 │
└─────┬──────┘   └─────┬──────┘   └─────┬──────┘
      │                │                │
      └────────────────┼────────────────┘
                       │
              ┌────────▼─────────┐
              │  access-manager  │
              │      :8080       │
              └──────────────────┘
```

## First run

```bash
docker compose -f compose.local.yaml up --build
```

That brings up:

| Service | Port | Credentials |
|---------|------|-------------|
| access-manager UI | http://localhost:8080 | login via Keycloak |
| Keycloak admin | http://localhost:8081 | `admin` / `admin` |
| Postgres | localhost:5433 | `ducklake` / `ducklake` |
| Garage S3 API | localhost:3900 | (S3 keys generated by access-manager) |
| Garage Admin API | localhost:3903 | bearer `local-dev-garage-admin-token` |

Two pre-seeded login accounts:

| Username | Password | Role |
|----------|----------|------|
| `admin@local` | `admin` | client role `admin` on `ducklake` |
| `student@local` | `student` | regular user |
| `student2@local` | `student` | regular user |

A `demo-dataset` bucket is created automatically by the `garage-bootstrap`
service so the Buckets list isn't empty on first login.

## Stopping

```bash
docker compose -f compose.local.yaml down            # keeps data
docker compose -f compose.local.yaml down -v         # nukes all volumes
```

## How the pieces fit (and why this took some thought)

### Keycloak + Spring issuer matching

Spring's `oauth2-resource-server` validates the `iss` claim of every JWT
against the configured `issuer-uri`. Keycloak emits `iss` from its
externally-visible URL, which from the browser's perspective is
`http://localhost:8081/realms/cloud`.

The catch is that the access-manager runs *inside* Docker, where
`localhost:8081` resolves to the access-manager itself, not Keycloak.
Two settings together solve it:

* `KEYCLOAK_ISSUER_URI=http://localhost:8081/realms/cloud` — what Spring
  expects to see in the `iss` claim. Used only as a string match,
  not fetched.
* `spring.security.oauth2.resourceserver.jwt.jwk-set-uri=http://keycloak:8080/realms/cloud/protocol/openid-connect/certs`
  — where Spring actually fetches signing keys (docker-internal DNS).

When `jwk-set-uri` is set Spring skips OIDC discovery, so the unreachable
`localhost:8081` URL is never followed by Spring itself.

### Frontend config

The original v1 frontend hard-coded the production Keycloak URL inline.
We added a `/api/config` endpoint that returns `keycloakBase` and
`clientId`; the frontend fetches it before mounting. This way the same
JS bundle works against KTH cloud, this local stack, and any future
environment without rebuilding.

### Garage bootstrap

A fresh Garage node has no cluster layout — until one node is assigned a
role and the layout is "applied", S3 calls fail. The one-shot
`garage-bootstrap` container talks to the Admin API, assigns the single
node a role, applies layout v1, and creates a `demo-dataset` bucket.
It's idempotent — safe to re-run on every `compose up`.

> Postgres is mapped to host port **5433** (not the default 5432) to avoid
> colliding with any other Postgres you might already have running.
> Inside the docker network the access-manager still reaches it on
> `postgres:5432`, so application config doesn't change.

## Faster dev loop (run access-manager on host)

The compose stack rebuilds the access-manager image on every `up --build`,
which is slow. For tighter iteration you can run the dependencies in
docker and the Spring app on the host:

```bash
# 1. Start only the deps
docker compose -f compose.local.yaml up keycloak postgres garage garage-bootstrap

# 2. Run access-manager from the project root with these env vars
SPRING_PROFILES_ACTIVE=local \
POSTGRES_HOST=localhost POSTGRES_DB=ducklake \
POSTGRES_ADMIN_USER=ducklake POSTGRES_ADMIN_PASSWORD=ducklake \
POSTGRES_PUBLIC_HOST=localhost \
KEYCLOAK_ISSUER_URI=http://localhost:8081/realms/cloud \
GARAGE_ADMIN_URL=http://localhost:3903 \
GARAGE_ADMIN_TOKEN=local-dev-garage-admin-token \
GARAGE_S3_ENDPOINT=localhost:3900 \
GARAGE_S3_REGION=garage \
mvn spring-boot:run
```

When running on the host the `jwk-set-uri` from `application-local.properties`
needs to point at `localhost:8081`, not `keycloak:8080`. The simplest way
is to also override it on the command line:

```bash
... -Dspring.security.oauth2.resourceserver.jwt.jwk-set-uri=http://localhost:8081/realms/cloud/protocol/openid-connect/certs
```

## Pointing at KTH cloud later

When KTH cloud lifts the NetworkPolicy restriction, deployment is just a
matter of using the original env vars from `.env.example` (production
values) — nothing in the codebase needs to change.
